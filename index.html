    Prerequisites

    • Kubernetes: v1.25+ with admin access
    • Helm: v3.8+ installed
    • Vault: Initialized HashiCorp Vault instance(s)

    ğŸƒâ€â™‚ï¸ Installation (60 seconds)

    1. Install the operator: 

      helm install vault-autounseal-operator \
  oci://ghcr.io/panteparak/vault-autounseal-operator \
  --namespace vault-system --create-namespace

    2. Create configuration: 

      cat <<EOF | kubectl apply -f -
apiVersion: vault.io/v1
kind: VaultUnsealConfig
metadata:
  name: my-vault
  namespace: vault-system
spec:
  vaultInstances:
  - name: vault-primary
    endpoint: https://vault.example.com:8200
    unsealKeys:
    - "base64-encoded-key-1"
    - "base64-encoded-key-2"
    - "base64-encoded-key-3"
    threshold: 3
EOF

    3. Verify it's working: 

      kubectl get vaultunsealconfigs -n vault-system
kubectl logs -n vault-system deployment/vault-autounseal-operator

    â“ FAQ

    How does the operator handle HA Vault clusters? The operator monitors individual pods in HA clusters using label selectors and automatically unseals new pods as they start, ensuring seamless failover.
    What happens if unseal keys are incorrect? The operator logs detailed error messages and continues attempting according to the reconciliation schedule. Status is updated in the VaultUnsealConfig resource.
    Can I manage Vault instances across namespaces? Yes! The operator supports cross-namespace monitoring with proper RBAC configuration.
    Is this production-ready? Absolutely. The operator includes comprehensive security, monitoring, error handling, and has been designed following production best practices.
    How do I migrate from the Python version? The Go version is a complete rewrite with the same API. Simply update your Helm deployment and your existing VaultUnsealConfig resources will work unchanged.
